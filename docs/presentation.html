Irvin Uyi Osazee · VUG/SEN/22/8386

The idea

What CI/CD is — and why

CI — Continuous Integration

Every code change is automatically built, linted and tested the moment it's pushed. Defects are caught in minutes.

CD — Continuous Deployment

Every change that passes CI is automatically packaged and shipped — to a registry and to the live server — with no manual steps.

The problem it solves: manual testing + "FTP-the-files-up by hand" is slow, error-prone and inconsistent. A pipeline makes it automatic, repeatable and gated — bad code can't reach production.

Stage 0 · the trigger

1.Git: the commit & the push

  • Commit = an immutable snapshot + a unique SHA hash (e.g. 89e729b…).
  • git commit records it locally — nothing has left your machine yet.
  • git push uploads commits to GitHub and moves the remote's main pointer. This push is the start button.

That same SHA travels all the way through — it even becomes a Docker image tag (vuna-calc:89e729b), so a running image traces back to the exact source commit.

Stage 1 · GitHub fires the workflow

2.Event → workflow → runner

The trigger

on:
  push:         [ main ]   # deploy
  pull_request: [ main ]   # test only

GitHub matches the event to .github/workflows/ci-cd.yml and creates a run.

The runner

Each job gets a fresh, throwaway Ubuntu VM (runs-on: ubuntu-latest).

Clean every time → reproducible. Nothing persists between jobs, so each re-runs checkout + npm ci.

Job 1 · the quality gate

3.ci — lint & test (5 gates)

✓ npm ci          # clean, lockfile-exact install (deterministic)
✓ npm run lint     # ESLint — static code-quality check
✓ npm test         # Jest — calculator tests + COVERAGE gate (70/80/80/80)
✓ npm run typecheck# tsc --noEmit — TypeScript type safety
✓ npm run test:unit# Vitest — 21 unit tests
✓ upload coverage  # saved as a downloadable artifact

If any step exits non-zero, the job fails and docker + deploy are skipped. This is the gate — broken code can't ship.

Job 2 · containerise

4.docker — build & push to Docker Hub

Multi-stage Dockerfile

FROM node:22-alpine AS build
... npm ci · lint · test · build
FROM nginx:alpine AS production
COPY --from=build /app/dist .

Final image = nginx + built files only (~small, secure — no Node/tests).

Build → login → push

setup-buildxlogin (secrets)build-push

Pushes irvinuyi/vuna-calc:latest + :<sha>.

Runnable anywhere: docker run -p 8080:80 …

Job 3 · go live

5.deploy — FTP to cPanel

✓ npm run build                 # Vite → produces dist/ (the deployable files)
✓ SamKirkland/FTP-Deploy-Action # upload dist/ over FTP
       server:    secrets.FTP_SERVER
       username:  secrets.FTP_USERNAME
       password:  secrets.FTP_PASSWORD
       local-dir:  ./dist/
       server-dir: ./public_html/    # cPanel's web root

The browser hits http://irvin.vudse26.cloud → cPanel serves public_html/index.html → the site runs in the browser.

How the jobs are wired

6.Orchestration: needs + if

            ┌─▶ docker   (needs ci · main+push only) → Docker Hub
ci ─ pass ──┤
            └─▶ deploy   (needs ci · main+push only) → cPanel
  • needs: [ci] → docker & deploy wait for tests to pass. The gate.
  • if: …main && push → they only fire on a real push to main; PRs run ci only.
  • docker & deploy then run in parallel with each other.
Security model

7.Secrets & hardening

SecretUsed by
FTP_SERVER / USERNAME / PASSWORDdeploy
DOCKERHUB_USERNAME / TOKENdocker
  • Encrypted by GitHub, masked in logs, never in the repo.
  • permissions: contents: read — least privilege for the auto token.
  • Docker Hub uses a revocable access token, not a password.
  • Repo is private; .gitignore blocks .env, keys, node_modules.
The two questions they'll probe

8.Why FTP? Why Docker if it doesn't run?

Why FTP, not SSH/rsync?

The host is CyberPanel with SSH locked down — the website user is no-login (SFTP corrupts), the main user is a restricted shell (blocks keys/SFTP). FTP is a separate service that works.

Why build Docker if cPanel can't run it?

Shared hosting has no Docker daemon for my user. So Docker satisfies the manual's containerise + registry stage; the image is a real, runnable artifact on Docker Hub. The live site is the static files.

Put it together

9.One commit, end to end

  1. commit → SHA recorded locally.
  2. push → GitHub main advances; a run starts.
  3. ci (VM A): checkout → npm ci → lint → Jest → typecheck → Vitest. ✓ ~30s
  4. ci passing unblocks docker + deploy (parallel).
  5. docker (VM B): build multi-stage image → push :latest + :sha. ✓
  6. deploy (VM C): vite build → FTP dist/ → public_html. ✓
  7. cPanel serves the new files → live site updated. Under a minute, automatic.
Robustness

10.What blocks what

If this fails……then
Lint / a test / low coverage / typecheckci fails → docker + deploy skipped
Docker Hub logindocker fails; deploy still runs (independent)
FTP auth (530)deploy fails; site unchanged
Push to a non-main branchonly ci runs — by design

Key property: the test gate protects production — deploy needs ci, so broken code cannot deploy.
